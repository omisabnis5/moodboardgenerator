## 0b. Change request v2 (2026-07-17)
User feedback after the first delivery. Adds/refines requirements; delivered as **PR-6**.

- **CR-1 — Full-width layout.** The app currently caps at a centered ~1080px column, leaving large empty side margins on desktop and feeling cramped. WHEN viewed on a wide screen, the app SHALL make effective use of the full width — a two-pane layout (selection controls beside the results) on large viewports, stacking to one column on narrow ones. → new **FR-15**, revises §6.
- **CR-2 — Reliability: no partial failures.** Users still saw 2 of 4 boards error out. The generation SHALL be hardened so that, under normal provider conditions, all four boards succeed — via bounded concurrency (a request queue) plus staggered starts and exponential-backoff retries. → revises **FR-9**, **NFR-Reliability**.
- **CR-3 — Stronger per-board variation.** The four boards SHALL be visibly, meaningfully different directions (distinct camera angle, lighting/time-of-day, and styling emphasis), not near-duplicates, so the user gets genuinely varied options. → revises **FR-8**.
- **CR-4 — Decorative artifacts in the rooms.** Generated rooms SHALL appear richly styled with decorative artifacts (framed art, ceramics/vases, plants, books, textiles/rugs, ornaments) so boards read as finished, styled spaces. → new **FR-16**, revises prompt in §8.

### Revised / new requirements
- **FR-8 (revised):** The 4 boards SHALL be four **independent directions** that differ visibly in composition, lighting, and styling emphasis (driven by 4 distinct seeds **and** 4 distinct prompt-variation phrases), not four near-identical images.
- **FR-9 (revised):** WHILE generating, the system SHALL limit in-flight image requests to a small concurrency (queue), stagger request starts, and auto-retry transient failures with exponential backoff, so all four boards normally succeed; a second concurrent Generate is still prevented.
- **FR-15 (new):** WHEN the viewport is wide (≥ ~1024px), the system SHALL present a two-pane layout — selection controls and the results grid side by side, using the full content width — and SHALL collapse to a single stacked column on narrow viewports.
- **FR-16 (new):** The generation prompt SHALL request richly styled decorative artifacts in the room (e.g. framed wall art, ceramic vases, potted plants, books, woven textiles/rugs, ornamental objects) so boards depict fully-styled spaces.
- **NFR-Reliability (tightened):** Target all-4-success under normal provider availability. **Finding (2026-07-17):** Pollinations gates browser-origin requests to ~2 per short window (later requests return 403 → ORB-blocked at the `<img>`), with a cooldown longer than the retry budget; server-side requests are not gated. Concurrent/staggered starts still lost 2 of 4. **Resolution:** default to **concurrency 1 (fully sequential)** generation — each request starts only after the previous settles (~8–12s apart), which stays under the limit and yields all-4-success (verified live). Kept exponential-backoff auto-retries (base 5s, cap 30s, ≤5) as a safety net. Total wall-clock ~40–50s for four boards — an accepted trade-off for reliability. Tuning (concurrency, stagger, backoff, timeout, retries) is overridable at runtime via `window.__MB_CONFIG__` so E2E runs fast/deterministic. A keyed provider behind a proxy would remove the gating and allow parallelism later.

### New acceptance criteria
- [ ] **AC-12:** On a wide viewport, controls and results render side by side using the full width; on a narrow viewport they stack — verified by **Playwright** (assert two-pane vs stacked via layout/box checks) + manual screenshot.
- [ ] **AC-13:** The four boards request four distinct prompts (distinct seeds **and** distinct variation phrases) — verified by **unit test** on the variation set + **Playwright** capturing 4 distinct request URLs.
- [ ] **AC-14:** The generated prompt includes decorative-artifact language — verified by **unit test** on `buildPrompt`.
- [ ] **AC-15:** With the provider healthy, a Generate yields four ready boards (no error tiles) under the bounded-concurrency queue — verified by **Playwright** (stubbed provider, all four `ready`) + a manual live check.

## 0. Resolved decisions (from clarify — 2026-07-16)
- **Q1 → AI photoreal image generation, keyless & free.** Boards are generated by a **text-to-image API** from a prompt built from the selections. Provider = **Pollinations.ai** (`https://image.pollinations.ai/prompt/<prompt>?width&height&seed&model=flux&nologo=true`): free, **no API key**, returns a photoreal JPEG directly, and a `seed` parameter yields distinct variations. Verified reachable (HTTP 200, on-brief traditional-pastel kitchen returned).
- **Architecture → client-only.** Because the provider needs **no secret**, image requests are made **directly from the browser**; there is **no backend/server proxy** and no hosting that must run server code. The provider is abstracted behind a small `ImageProvider` interface so a paid/keyed provider can be swapped in later (which *would* then require a proxy).
- **Q4 → React + Vite + TypeScript**, static build, deployable to any static host.
- **Q2 → provider Pollinations (`flux` model); acceptable cost = $0.** No account required.
- **Q3 → fixed palette hex sets defined** (see §8) — used both for on-board swatch rows and to steer the prompt's color language.
- **Q5 → per-image timeout 45s; no auto-retry, user-driven Retry (FR-11); rate guard:** Generate is locked while a batch is in flight; the 4 images are requested per batch (bounded concurrency).
- **Q6 → download per board only** for v1; export-all deferred (kept a non-goal).
- **Q7 → on selection change after a generation, keep the existing boards but mark them stale** ("selections changed — regenerate"); the next Generate replaces them. On error, prior boards are not shown during loading; the error state offers Retry.
- **Q8 → four independent directions** (distinct seeds + light prompt-phrase variation), not one board + 3 variants.
- **Q9 → neutral, clean branding** for v1 (no firm logo supplied).
- **Q10 → static hosting** (client-only bundle).

## 1. Summary
A single-page web app for an interior-design firm that lets a user pick a **room type**, a **design style**, and a **color palette** from predefined options, then generate **four distinct mood board images** that reflect the chosen combination (e.g. Kitchen + Traditional + Pastels → four fully-furnished traditional-pastel kitchen mood boards). The four boards give the designer a spread of on-brief visual directions to choose from and share with clients.

## 2. Goals
- Let a user express a design brief in three clicks (room type, style, palette) and get 4 mood boards without typing.
- Every generated board is visibly consistent with all three selected parameters.
- The four boards are meaningfully distinct from one another (varied compositions/directions), not four copies.
- First-time-to-result is fast and the flow is obvious with no instructions.
- Works on desktop and mobile browsers, keyboard accessible.

## 3. Non-goals
- User accounts, authentication, saved projects, or multi-user collaboration.
- Editing/rearranging a board's contents by hand (drag-drop board editor).
- E-commerce, product sourcing, or price/vendor data.
- Free-text prompt input or parameters beyond the three predefined groups.
- Persisting generated boards to a server database (beyond optional local download).
- 3D room rendering or floor-plan layout.

## 4. Users & scenarios
- **Interior designer** wants to quickly produce several on-brief visual directions for a room so that they can present options to a client and pick a direction.
- **Design-firm account manager** wants to generate and download boards to drop into a client deck so that proposals look polished with minimal effort.
- **Prospective client (self-serve)** wants to explore how a room could look in a chosen style and palette so that they can articulate their taste to the firm.

## 5. Functional requirements
> EARS-style, testable. Each has a stable ID. The generation *mechanism* (AI model vs. curated collage) is an Open Question (Q1); requirements below are written to hold regardless of which mechanism is chosen, and are marked where the answer changes detail.

- **FR-1:** WHEN the app loads, the system SHALL display three labeled selection groups — Room Type (8 options), Style (8 options), and Color Palette (6 options) — rendering each option's `label` and `description` from the provided option lists.
- **FR-2:** The Room Type group SHALL offer exactly: Living Room, Dining Room, Bedroom, Children's Room, Kitchen, Bathroom, Home Office, Entryway.
- **FR-3:** The Style group SHALL offer exactly: Bohemian, Modern, Traditional, Contemporary, Industrial, Scandinavian, Mid-Century Modern, Farmhouse.
- **FR-4:** The Color Palette group SHALL offer exactly: Neutral Warm, Cool Blues, Earth Tones, Monochrome, Jewel Tones, Pastels.
- **FR-5:** WHEN the user selects an option within a group, the system SHALL make that the single active selection for that group (single-select) and visually indicate the selected state.
- **FR-6:** WHILE any of the three groups has no selection, the system SHALL keep the Generate button disabled and indicate that all three choices are required.
- **FR-7:** WHEN all three groups have a selection and the user activates Generate, the system SHALL produce exactly **4** mood board images that reflect the selected room type, style, and color palette.
- **FR-8:** The 4 generated boards SHALL be distinct from one another (different composition/variation), not four identical images.
- **FR-9:** WHILE generation is in progress, the system SHALL show a loading/progress state and prevent a second concurrent generation.
- **FR-10:** WHEN generation completes successfully, the system SHALL display the 4 boards in a responsive grid, each labeled with the selected room type / style / palette.
- **FR-11:** WHEN generation fails (mechanism error, network error, or timeout), the system SHALL show an error state with a human-readable message and a Retry action that re-runs generation with the current selections.
- **FR-12:** WHEN the user changes any selection after a generation, the system SHALL allow generating again and replace the previous results with the new set.
- **FR-13:** The system SHALL let the user download/save each generated board image individually. *(Export-all is Open Question Q6.)*
- **FR-14:** WHEN a color palette is selected, the generated boards SHALL visibly incorporate that palette's colors (e.g. as a swatch row and/or dominant tones). *(Exact rendering depends on Q1.)*

## 6. UI / UX (if applicable)
No mockups were provided; screens/states are derived from the requirements. Single-page layout.

**Screens / regions**
- **Selection panel** — three groups (Room Type, Style, Color Palette), each a set of selectable option cards showing label + description. Palette options additionally preview their colors.
- **Action bar** — Generate button (primary), plus a summary of the current selection ("Traditional · Pastels · Kitchen").
- **Results area** — 2×2 grid of four mood boards; each board has a label and a download control.

**States**
- **Initial / empty** — no selections; Generate disabled; results area shows a placeholder/prompt to make selections.
- **Partial selection** — one or two groups chosen; Generate still disabled with a hint of what's missing.
- **Ready** — all three chosen; Generate enabled.
- **Loading** — after Generate; four skeleton/placeholder tiles + progress indication; controls locked.
- **Success** — four distinct boards rendered in the grid, each labeled + downloadable.
- **Error** — generation failed; message + Retry; previous results (if any) handling per Q7.
- **Empty results after change** — changing a selection post-generation invalidates/marks stale the old boards (behavior per Q7).

**Interactions & accessibility**
- Each group behaves like a single-select radiogroup: options reachable and selectable by keyboard; selected state conveyed to assistive tech.
- Generate is a button with a clear disabled reason; loading state announced.
- Responsive: selection groups and 2×2 grid reflow to a single column on narrow viewports.

## 7. Non-functional requirements
> No constitution present; these are sensible defaults to be ratified via `/specs:constitution`.

- **NFR-Performance:** Initial page interactive within ~2s on a typical broadband desktop. UI selection feedback < 100ms. Generation perceived latency target depends on Q1: **curated/collage** boards SHALL render within ~1s; **AI image generation** SHALL show progress and target < ~30s for all four, with a defined timeout (Q5).
- **NFR-Accessibility:** Target WCAG 2.1 AA — full keyboard operability, visible focus, correct radiogroup/button semantics and ARIA state, color contrast ≥ 4.5:1 for text, and selection state not conveyed by color alone.
- **NFR-Security:** No user accounts or PII collected. The chosen provider (Pollinations) is **keyless**, so there is **no secret to expose** and no proxy required; the browser calls the image URL directly. The three selected parameters are constrained to the fixed option `value`s (union types) before being composed into a prompt — no free-text/user-injected content reaches the URL. No secrets in the repo. NOTE: if a keyed provider is later swapped in via the `ImageProvider` interface, a server-side proxy that reads the key from env becomes required — call this out at that time.
- **NFR-Reliability:** Generation failures are caught and surfaced as the Error state (FR-11), never a blank/crashed page. IF an external API is used, requests have a timeout and bounded retry; a partial result (fewer than 4 images) is handled gracefully.
- **NFR-Observability:** Client logs generation start/success/failure with the selected parameters (no PII). IF a backend exists, it logs upstream API latency and error codes. Minimal, privacy-respecting.
- **NFR-Compatibility:** Latest 2 versions of Chrome, Edge, Firefox, Safari; responsive for mobile/tablet/desktop.
- **NFR-i18n:** UI copy is in English for v1; option labels/descriptions come from the provided lists. Strings SHALL be kept in a single module to ease later localization; full localization is out of scope for v1.
- **NFR-Cost (conditional):** IF AI image generation is used, per-generation cost and a rate/spend guard are required (Q5) to prevent runaway spend from repeated Generate clicks.

## 8. Data & contracts
- **Domain types** (from the prompt): `RoomType`, `Style`, `ColorPalette` union types and `ParameterOption<T> = { value: T; label: string; description: string }`. The three option arrays (`ROOM_TYPE_OPTIONS`, `STYLE_OPTIONS`, `COLOR_PALETTE_OPTIONS`) are the source of truth for choices.
- **Palette color sets (Q3, fixed):** each `ColorPalette` maps to an ordered list of hex swatches, used for the on-board swatch row and to steer the prompt's color language:
  - `neutral_warm`: `#EDE4D3, #D9C6A5, #B79F86, #8C7B6B, #5E5245`
  - `cool_blues`: `#DDEBF2, #A9CCE3, #5B9BD5, #2E75B6, #1F4E79`
  - `earth_tones`: `#E8E0CF, #A3B18A, #E07A5F, #B5651D, #6B4226`
  - `monochrome`: `#FFFFFF, #D9D9D9, #A6A6A6, #595959, #1A1A1A`
  - `jewel_tones`: `#046A38, #0F52BA, #6A0DAD, #7D0633, #C9A227`
  - `pastels`: `#FDE2E4, #FAD2E1, #D8C2EF, #BEE1E6, #DCF3E8`
- **`ImageProvider` interface (client-side):** `generate(request): Promise<BoardResult>` where `request = { roomType, style, colorPalette, seed, variationHint }` and `BoardResult = { imageUrl: string }`. Default implementation = `PollinationsProvider` building `https://image.pollinations.ai/prompt/<encoded prompt>?width=1024&height=1024&seed=<seed>&model=flux&nologo=true`.
- **Batch:** one Generate produces 4 board descriptors `{ id, imageUrl, seed, label, status: 'loading'|'ready'|'error' }` with 4 distinct seeds and light per-board `variationHint` phrasing (Q8).
- **Prompt composition:** deterministic function `buildPrompt({roomType, style, colorPalette, variationHint})` → e.g. *"a {style.label} {roomType.label} interior, fully furnished and styled, {colorPalette} color palette ({hex list} tones), professional interior design photograph, natural light, high detail — {variationHint}"*. Pure/unit-testable.
- **API contract:** None — no backend. All requests are direct browser GETs to the provider URL.
- **Migrations / backfills:** None (no database). Downloads are client-side (fetch image → object URL → `<a download>`).

## 9. Dependencies & assumptions
- New project, empty repo; no existing code to integrate with.
- Node 20 and npm available locally.
- `origin` GitHub remote exists; **`gh` CLI is NOT installed** in this environment, so the specs `execute` PR-open/merge-to-`main` step cannot run automatically — commits/pushes must be done directly or `gh` installed. (Assumption to confirm.)
- Assumes the three provided option lists are final and complete for v1.
- Generation depends on the **Pollinations.ai** public endpoint being reachable at runtime (no account/key). If it is down or slow, the app surfaces the Error state + Retry (FR-11). Provider is swappable via `ImageProvider`.
- Because generation requires network access to a third-party service, **automated Playwright verification will stub/intercept the image requests** (serve a local fixture image) so tests are deterministic and offline; a separate manual/live check confirms real generation.
